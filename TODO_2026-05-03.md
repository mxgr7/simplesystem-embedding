## Critical gaps / discrepancies

1. **ArticleId format no longer matches next-gen / spec**
   - Issues/spec require `{friendlyId}:{base64Url(articleNumber)}`.
   - Next-gen `ArticleId.java` uses `FriendlyId.toFriendlyId(...)`.
   - Current code/tests now use `{vendor_uuid_dashed}:{base64Url(...)}`.
   - `indexer/friendly_id.py` is gone, despite I1 saying it landed.
   - This breaks the stated non-deviation contract and likely breaks any existing caller/parser or `articleIdsFilter` values generated by legacy.

2. **Indexer hash / embedding identity is inconsistent and currently failing tests**
   - `indexer/projection.py` now has `HASH_VERSION = "v2"` and hashes a new field set.
   - `indexer/duckdb_projection.py` still computes the old F9 hash over name/manufacturer/categories/eclass.
   - Issues/F9 docs still describe the old hash field set.
   - `TEICache.embed_articles()` renders article rows that do not contain the new hash/input fields (`description`, `categoryPaths`, manufacturer article fields), so cache misses may embed the wrong text.
   - Targeted pytest showed this is broken:
     - `tests/test_projection.py tests/test_tei_cache.py`: **7 failed**
     - DuckDB parity/aggregate tests: **5 failed, 6 errors**

3. **Uncommitted DuckDB preprojection work is broken**
   - Working tree is dirty:
     - modified `indexer/bulk.py`
     - modified `indexer/duckdb_projection.py`
     - untracked `scripts/build_offer_projected.py`
   - `_build_flat_rows_sql()` now references `finalized` without including `_FINALIZED_CTE_SQL`, causing DuckDB errors.
   - Aggregate parity also fails 100% on `article_hash`.

4. **Legacy catalog/price scoping is not implemented like next-gen**
   - Next-gen `SearchArticleQueryBuilder.coreQuery()` always applies:
     - `OfferFilterBuilder`: catalog scope using `catalogVersionIdsOrderedByPreference` or `closedCatalogVersionIds`
     - `PriceListFilterBuilder`: `sourcePriceListIds` + top-level `currency`
   - Current ftsearch:
     - only filters catalog when `closedMarketplaceOnly=true`
     - only emits price-list/range prefilter when `priceFilter` exists
     - does not emit a standalone currency prefilter
   - This contradicts `issues/article-search-replacement-spec.md` §4.3 and F8’s “emit when field is set” text.

5. **ACL → ftsearch default target is wrong for F9**
   - ACL default collection is `"articles"` (`acl/clients/ftsearch.py`, `acl/app.py`).
   - Dedup search endpoint expects URL collection to be the **offers** alias; article collection is separate env.
   - Tests assert calls to `/articles/_search`, masking the issue.
   - `playground-app/compose.yaml` also does not set `USE_DEDUP_TOPOLOGY=1`.

6. **Compose/API-key mismatch**
   - `playground-app/compose.yaml` sets `search-api` `API_KEY`.
   - ACL client does not send `Authorization` / `X-API-Key`.
   - Spec says no auth on ACL→ftsearch, so either compose should not set `API_KEY` for this path or ACL needs configured forwarding.

7. **Error envelope shape does not match next-gen**
   - Next-gen OpenAPI and `RestApiControllerAdvice` use:
     ```json
     "details": [{"field": "...", "message": "..."}]
     ```
   - ACL currently returns:
     ```json
     "details": ["field=...: ..."]
     ```
   - `acl/openapi.yaml` also declares `items: string`.
   - A4 issue text was updated to the wrong string-array shape.

8. **A6 acceptance tests are too weak / partially vacuous**
   - No per-filter, per-sort, per-aggregation ACL acceptance yet.
   - Captured traffic harness exists, corpus does not.
   - `test_articleid_format_round_trips` only checks `":"`; with zero hits it passes without checking anything.
   - Because no-query/no-filter can return empty, explain/articleId assertions can silently skip all articles.

9. **eClass hierarchy aggregation is likely wrong for real legacy codes**
   - Current `_eclass_depth()` derives depth from digit length.
   - Next-gen `EClassGroup` uses 8-digit zero-padded base-100 levels: e.g. `23000000`, `23110000`, `23110100`, `23110101`.
   - Tests use toy unpadded values like `23`, `2317`, so real padded hierarchy behavior is not covered.
