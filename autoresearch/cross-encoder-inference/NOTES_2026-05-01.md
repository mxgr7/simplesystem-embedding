## 2026-05-01 — pretokenized offer cache: −176 ms p95 (1712 ms total)

The cross-encoder input is `[CLS] query [SEP] offer [SEP]`. The offer half
is independent of the query, so it can be word-piece tokenized once at
catalog-load time and reused across requests. A 4.2 MB `.npz` sidecar
stores the offer-side token arrays for every offer in the bench fixture;
at request time we numpy-assemble the pair tokens directly into the
existing per-chunk pinned buffers (no rendering, no offer-side tokenize).

### Implementation

- `build_offer_tokens.py`: renders + Rust-tokenizes (no special tokens, no
  padding) every offer in the bench fixture, truncates each to ≤
  max_pair_length − 3, writes
  `fixture_2000x512_offer_tokens.npz` with offer_token_ids (int32, N × T),
  offer_lengths (int32, N), offer_ids (string, N).
- `verify_pretokenized.py`: byte-equality check between HF
  `tokenizer(query, offer_text, ...)` and our numpy assembly. All
  2000/2000 input_ids, attention_mask, token_type_ids match.
- `Reranker.rerank_pretokenized(query, offer_ids)`: query encode (one
  short call), cache lookup, numpy assembly into pinned buffers, same
  pipelined per-chunk forward as `_pipelined_forward`. Wrapped in
  `torch.inference_mode()` — see "OOM" note below.
- `RerankerConfig.offer_tokens_path` / `SERVE_OFFER_TOKENS_PATH` env: when
  set, server loads the .npz at startup and exposes `/rerank_pretokenized`
  (request body: {query, offer_ids, threshold?, top_k?}).
- `bench_rerank.py --pretokenized`: posts offer_ids only (request payload
  drops 4 MB → 30 KB).

### OOM landmine

The first attempt at the pretokenized path OOM'd the 24 GB 4090 during
warm 1 (`empty_strided_cuda((128, 512, 3072), …)` — the FFN intermediate
at 384 MB per layer × 12 layers × 16 chunks = ~73 GB if all alive
simultaneously). Root cause: the pretokenized worker is so fast (~10 ms
to assemble vs ~50 ms per-chunk tokenize) that all 16 GPU forwards get
queued back-to-back; without `torch.inference_mode()`, the autograd
graph for each chunk's forward holds the activations alive until the
output tensor is consumed (or could be backwarded). Fix:
`with torch.inference_mode(): ...` around the chunked forward loop.
inference_mode disables autograd version-tracking + graph construction,
so activations free between chunks. The existing `_pipelined_forward`
path didn't need this because its slower per-chunk worker naturally
spaced the GPU forwards.

### Production deployment

The offer catalog is finite; offer_token arrays can be precomputed
offline and cached. ~10 M offers × ~256 tokens × 2 bytes int16 ≈ 5 GB;
fits in server RAM. The serving API would accept either offer dicts
(current `/rerank`) OR offer_ids (cached `/rerank_pretokenized`). The
.npz format used here generalizes; the only changes for production
catalog size: chunk the catalog by hash for multi-process loading, and
expose a refresh endpoint when the catalog changes. Deployment shape is
explicitly out-of-scope for this PR — the bench validation is the
deliverable.

## 2026-05-01 — pinned-buffer h2d: −133 ms p95 (1898 ms total)

After encode_batch_fast cut tokenize 1716 → 817 ms and the description
char-cap shaved another 87 ms, the new top stage was sum_h2d at 770 ms
across 16 chunks — anomalously high because `tensor.to(device,
non_blocking=True)` on a *pageable* source synchronously page-locks the
buffer per call before transferring. The "dispatch" timer was actually
measuring the page-lock kernel call.

Pre-allocate double-buffered pinned host tensors (chunk_size ×
max_pair_length × int64 × 2 slots × 3 tensors = ~384 KB) at __init__.
Worker fills slot (i+1)%2 in-place via `tensor.numpy()` view. Two slots
suffice: worker writes slot (i+1)%2 while GPU is consuming slot i; by
the time worker hits slot (i+2)%2 = slot i again, GPU has long
released its dependency on it.

Bench (n=30): p50 2014→1889 ms, p95 2031→1898 ms (-133). sum_h2d 770→3
ms. The calibration step (softmax + cuda→cpu sync) absorbed some
residual GPU work — went 65→658 ms — but that's GPU work that was
previously hidden inside the per-chunk h2d-stall. Net pipelined_forward
+ calibration dropped 139 ms.

Cumulative session journey on the 12L distill ckpt:
- pre-session p95: **3107 ms** (distill-long-binary keep)
- + encode_batch_fast tokenize: 2118 ms (-989)
- + description char cap (4096): 2031 ms (-87)
- + pinned-buffer h2d:           **1898 ms** (-133)
- session total: **−1209 ms (-39%)**, target 2500 met by 602 ms

Quality unchanged throughout (same model ckpt, same Rust BPE,
byte-identical token IDs); f1_exact = 0.9081 on full val.

Next-largest stages: sum_tokenize ≈ 590 ms, sum_render ≈ 450 ms,
calibration ≈ 660 ms. The pipeline bubble is now ≈ 580 ms (worker
prep > GPU forward by ~36 ms/chunk × 16). Closing the bubble
further requires either parallelizing the per-pair render across
threads (Python loop holds GIL, but inner regex calls release it —
modest win) or chunk-size tuning at the new prep/forward ratio.

Chunk-size sanity check: re-bench with chunk=256 (vs default 128)
gave p50=1938, p95=2025 (slightly *worse*), peak_vram 3.2→4.1 GB.
Per-chunk numbers: GPU dispatch 169 ms (vs 265 at chunk=128 — fewer
launches), sum_tokenize 465 ms (vs 588 — fewer calls), sum_render
~unchanged, bubble 649 ms (vs 581 — bigger per-chunk worker gap).
chunk=128 stays optimal under the new pinned-buffer regime.

## 2026-05-01 — encode_batch_fast tokenize: −989 ms p95, target met

The 12L distill keep at p95=3107 ms had 575 ms over the 2500 target. Earlier
attempts via smaller backbone (8L base, no win) and bnb8 GPU INT8 (slower) failed.
Sub-stage profiling (SERVE_DEBUG_TIMING=1 env toggle in `inference.py` and
`server.py`, instrumented per-chunk inside `_pipelined_forward`) revealed:

| stage              | before  | comment                                        |
|--------------------|--------:|------------------------------------------------|
| body_read          |  50 ms  | negligible                                     |
| json_parse         |  43 ms  | orjson, negligible                             |
| sum_render         | 832 ms  | 16 chunks of `_render_offer_fast`              |
| sum_tokenize       |**1716 ms**| 16 chunks of HF `tokenizer(...)` ← bottleneck |
| sum_h2d            |   8 ms  | negligible                                     |
| GPU dispatch       | 382 ms  | 16 chunks of `model()` calls                   |
| bubble             |2024 ms  | wall − GPU; main thread idle on prep           |
| calibration        |  66 ms  | softmax + cuda→cpu sync                        |
| response build     |   3 ms  | dataclass + dict + orjson                      |
| **wall**           |**2975ms**|                                               |

Tokenize was 60% of the request. Offline microbench on the same fixture showed
`tokenizer(...)` (HF Python wrapper) costs ~1480 ms while
`tokenizer._tokenizer.encode_batch_fast(pairs)` (direct Rust) costs ~440 ms —
**a ~1000 ms cliff in the Python wrapper alone** (arg validation, per-call
padding logic, list-comp tensor build).

Fix in `_pipelined_forward`: configure `enable_padding` + `enable_truncation`
once on the underlying `Tokenizer` at __init__, then per-chunk call
`encode_batch_fast(pairs)` directly. Assemble tensors via pre-allocated
`np.empty(...)` buffers + `torch.from_numpy(...)` (zero-copy).

After:

| stage              | before  | after  | Δ      |
|--------------------|--------:|-------:|-------:|
| sum_render         | 832 ms  | 842 ms |  +10   |
| sum_tokenize       |1716 ms  | 817 ms | **−899** |
| sum_h2d            |   8 ms  | 127 ms |  +119  |
| GPU dispatch       | 382 ms  | 336 ms |   −46  |
| bubble             |2024 ms  |1194 ms |  −830  |
| **p95 (n=30)**     |**3107ms**|**2118 ms**|**−989**|

The 2118 ms p95 clears the 2500 ms target by 382 ms. h2d went up because the
new path does `torch.from_numpy` + 3× `.to()` calls (was 1 `.to()` covering all
3 tensors), but the cost is dwarfed by the tokenize savings. Quality
unchanged: same model ckpt, same Rust BPE, same padding config → byte-identical
token IDs to the HF wrapper. Distill-long ckpt still scores `f1_exact=0.9081`
on full val.

**Lesson:** when the GPU forward is short relative to CPU prep (here forward
≈ 380 ms, prep ≈ 2550 ms), GPU-side optimizations (smaller backbone, INT8,
fp8, TRT) only shift which stage is critical-path; they do not shrink wall
clock. Profile per-stage *first*; only then pick a fix. The `bnb8` and 8L
attempts in this branch wasted ~30 minutes of compute and added negative
value because the timings weren't measured first.

**Next top stage:** sum_render = 842 ms is now the largest single stage.
Pipeline bubble = 1194 ms is the wall-clock waste from prep > forward; closing
it requires either making prep even faster or letting GPU forward absorb more
of the cycle. Worth profiling inside `_render_offer_fast` (HTML clean,
per-field f-string, normalize regex) before touching it.

## 2026-05-01 — quality floor pivot: 4-class macro_f1 → binary f1_exact

The 4-class macro_f1 ≥ 0.740 floor was replaced with a single binary floor:
**`val/cls/f1_exact ≥ 0.880`** on canonical full val. Reasoning:

1. **Downstream contract** — only `p_exact_calibrated` (the temperature-scaled
   softmax probability for the Exact class) is consumed by callers. The
   Substitute/Complement/Irrelevant probs and `predicted_label` are
   compatibility fields that downstream ignores. Gating on a metric that
   doesn't reflect what production uses is misaligned by construction.
2. **Complement-class collapse is structural, not budgetary** — across 1h,
   2h, and 4h distill runs the Complement F1 plateaus at 0.42–0.45 regardless
   of training time. The class is 1.6 % of train and 1.8 % of val, and the
   teacher itself only hits ~0.65 on it; any small-student distillation
   inherits a hard ceiling that no amount of compute breaks. Macro_f1
   (0.740 floor) was 25 % weighted on this single capped class — so the
   gate was effectively "did Complement converge", not "did the model
   learn the task downstream cares about".
3. **Headroom math** — soup teacher hits f1_exact ≈ 0.91, distilled student
   hits ≈ 0.90, floor at 0.88 leaves ~0.02 of compression headroom (room
   for INT8 / fp8 / further distillation) without trivially passing every
   reasonable candidate.

The 4h finalized gelectra-base distill (rare-penguin-951, p95 3107 ms) failed
the old macro floor by 0.016 (driven entirely by complement F1 0.4122) but
clears the new f1_exact floor easily — flipping it from `discard` to a new
production keep candidate at 3107 ms p95, the fastest model that holds quality.

The 0.880 floor is **on canonical full val**. The 10k subset has noise budget
≈ ±0.003 on the binary metric (Exact is ~52 % of val, much more stable than
macro_f1's ±0.013) — borderline calls (within 0.005) get a full-val confirm
before any commit/discard.

Pre-2026-05-01 results.tsv rows have `val_micro_f1` / `val_macro_f1` columns
preserved (informational), with `f1_exact=X.XXXX` appended retroactively to
descriptions where it was cheap to compute from existing checkpoints.

### Where the rest of the gap to 2500 ms is

For the production gelectra-large path: forward alone is ~5500 ms (24L 1024H
encoder, S=512, B=128 × 16 chunks). Hardware-bound on 4090 — only smaller
backbones get under 2500 ms.

For the distilled gelectra-base path: forward 1875 ms + prep 800 ms + HTTP/JSON
~250 ms. Pipelining the prep with forward gave only ~50 ms savings (vs the
predicted ~700 ms) because chunk=128 already overlapped CPU-prep with
GPU-forward implicitly via the Python interpreter's natural concurrency on
non-GIL operations. The remaining gap requires either a smaller student
(8L/6L) or a fundamentally faster forward backend (fp8 / TRT-fused SDPA).

### Recommended next steps

1. **Distill to 8L pruned gelectra-large** — preserves H=1024 capacity (so
   Complement learning is more likely to stick) while cutting layer count
   by 3×. Latency target ~3500/3 ≈ 1200 ms forward, well under 2500 ms total.
2. **Re-export ONNX with optimum's bertology fuser** — manual TRT engine
   build off the current legacy ONNX didn't get fused-MHA; optimum's
   transformers exporter produces a graph that TensorRT's built-in MHA
   plugin can fold.
3. **Class-balance the train sampler for Complement** — the persistent
   0.40-0.50 F1 plateau on Complement is downstream of its 1.6% prevalence
   in train; an oversample (e.g., 5×) at constant α=0.7 should push macro
   above floor without disturbing the rest.

## 2026-05-01 — fast renderer + fast normalize: −1750 ms p95

The Jinja-based RowTextRenderer.render_offer_text loop costs ~1.3 ms/offer
(≈2.5s for 2000 offers) — *bigger* than the GPU forward on the small student.
Replaced with a hand-rolled f-string builder. Then went further: the inner
`normalize_text` runs 3 regex subs unconditionally, but for already-clean
text (production traffic + bench fixture) all three are no-ops. Adding cheap
`in` pre-checks before each sub cut renderer cost from ~1450 ms → ~830 ms
on the fixture. Output byte-identical to the slow path (verified 0/2000
mismatches). Saves a cumulative −2.3s across both paths.

Subtle bug uncovered while making this byte-equal: `embedding_train.text._BR_RE`
uses `r"(?i)<br\\s*/?>"` which is a literal `\s` (backslash + s), not the
regex metachar — so `<br />` never matches and falls through to `_TAG_RE`,
becoming a space rather than a newline. The fast renderer mirrors this
buggy behavior for output-equivalence; *fixing* the BR regex would also
require re-evaluating the floor since training-time text would change.

## 2026-04-30 — distillation hyperparameter sweep findings (gelectra-large → gelectra-base)

20-min sweeps with `val_check_interval=0.05`, `limit_val_batches=0.1`:

|-------|------|-----|-----|-----|-------|-----------|--------|-------------|-------------|---------|
| iter1 | 3e-5 | 2.0 | 0.7 | F   | 0     | none      | 0.05   | 0.8602 (full @ 0.66ep) | 0.7282 | floor-fail @ 1h |
| iter2 | 5e-5 | 1.5 | 0.5 | T   | 0     | none      | 0.05   | 0.8464 @ 0.25ep | -    | killed |
| A v2  | 1e-4 | 2.0 | 0.7 | F   | 0     | none      | 0.03   | **collapsed** | -      | LR too high |
| B     | 5e-5 | 2.0 | 0.7 | F   | 0     | none      | 0.05   | learned then **collapsed** at step ~2000 | - | LR 5e-5 unstable |
| **C v2** | **3e-5** | **2.0** | **0.7** | **F** | **0** | **1.0** | **0.1** | **0.8535** | **0.6821** | **best — escalated** |
| D     | 3e-5 | 2.0 | 0.7 | F   | 2.0   | 1.0       | 0.10   | 0.8444 (declining) | 0.6573 | focal hurt |

Key findings:
- **lr=3e-5 is the stable sweet spot.** lr=5e-5 collapses to "predict-all-Exact"
  after ~2000 steps; lr=1e-4 collapses immediately during warmup.
- **gradient_clip_val=1.0 is cheap insurance** against late-training collapse.
- **Class weights and focal_gamma both hurt at 20-min budget.** Auto-computed
  Complement weight (15.6×) overwhelms majority class learning on cold
  student; focal_gamma=2.0 oscillates. Plain CE+KL beats both.
- **Complement plateaus around F1=0.45** across all distill runs (1h, 2h,
  4h budgets). It's the bottleneck for macro_f1 hitting the floor; longer
  training alone does not help. A class-balanced sampler is the obvious
  next experiment.
- **MLflow metric history URL-encoding gotcha**: `--data-urlencode metric_key=val/cls/X`
  silently returns empty for some keys; use `-d 'metric_key=val%2Fcls%2FX'`
  (manual %-encoding).

## 2026-04-30 — backends that didn't help

- **ONNX RT CUDA EP** (no fused SDPA): p95 12740 ms — 2× slower than
  torch.compile + max-autotune for this small encoder.
- **ONNX RT TRT EP** (libnvinfer.so.10 missing initially; provisioned via
  `uv add tensorrt`): p95 4792 ms with chunk=256, 4262 ms with chunk=250
  (1 unique shape). Still slower than torch path. The legacy ONNX exporter
  doesn't emit a graph that TRT's MHA plugin can fold; optimum's exporter
  would likely be required.
- **ONNX RT INT8 dynamic** on CUDA EP: 172 Memcpy nodes inserted (no INT8
  matmul kernels for transformer ops on CUDA EP) — eval ran 17 min and
  stalled. INT8 only viable through TensorRT EP with static calibration.
- **6L pruned gelectra-large** student: 20-min sweep micro 0.78, macro
  0.53, trajectory declining at end. Too aggressive a cut; 8L is the next
  obvious checkpoint.
- **fp16 weights vs bf16**: essentially identical on Ada (p95 5204 vs 5393).
- **Threaded renderer** (ThreadPoolExecutor(4)): p95 +2000 ms. GIL contention
  + thread setup outweighs concurrent regex scan in this size range.

## 2026-04-30 — 4090 baseline anchor

Pure gelectra-large + bf16 autocast + SDPA + chunk=256: **9234 ms p95** for
2000 offers at S=512, single client. H100 reference is 681 ms median —
4090 is 13× slower, much more than the FLOPS ratio (~2×) would predict.
Encoder is effectively compute-bound at our shape. Pure precision tricks
(fp8) and architectural cuts (distill, prune) are the only big remaining
levers.
